2026-04-04 · Disease Briefing Skill Assessment · 5 advisors, 5 peer reviews

Is the /disease-briefing skill complete enough for production use? What gaps exist?

The Recommendation

Not production-ready. Ship in two weeks, not today.

Three P0 defects would generate immediate user complaints or silent data loss. These are not architectural problems — the Executor estimates ~2 hours for all three. Fix the P0s, declare the scope (triage tool vs analysis tool), and ship.

The One Thing to Do First

Add a subscription pre-check at Step 1 with an explicit early exit and clear error message. This is the highest-severity user-facing failure. Users without premium accounts burn 30+ seconds before discovering they can't access the content.

P0 Fixes (~2 hours total)

1. Subscription Pre-Check

Premium feature doesn't check subscription upfront. User waits 30+ seconds then gets nothing. Add early exit at Step 1 if search returns error/empty.

2. Section ID Extraction Undocumented

SKILL.md doesn't explain how to get section IDs from Step 2 for use in Step 3. Silent failure produces incomplete report with no signal. Add explicit extraction instructions.

3. 500-Character Truncation Hardcoded

Delivers ~5% of actual briefing content. No override option. Parameterize the limit (default 5000, configurable). 3-line code change.

Blind Spots the Council Caught

Cascade Failure

If Step 2 section ID extraction fails silently, Step 3 runs on garbage, and the user receives an incomplete briefing with no error signal.

No Testability

No mock path, no fixture, no dry-run. The skill cannot be validated without a live premium account.

User Task Undefined

The skill doesn't declare whether it's for triage (500 chars fine) or analysis (500 chars broken). Both user types will be disappointed differently.

Advisor Responses

The Contrarian

Most gaps are nice-to-haves. The 500-char truncation is deliberate for scannability. No disease name resolution needed — API does fuzzy matching. Cross-skill hints are sugar. Multimedia irrelevant for CLI. Ship the MVP — how many users even have premium subscriptions?

The First Principles Thinker

The skill delivers ~5% of actual briefing content (500 chars x 9 sections = 4,500 chars vs tens of thousands). No relevance validation on search results. Undocumented section ID extraction between steps. This is a teaser, not a briefing. Fix: full text or user-selected section expansion.

The Expansionist

Missed opportunity on premium API. Uses 1 of 4 Drug Design endpoints. Needs: full text option, CSV export, manifest, subscription pre-check, multimedia metadata, cross-reference with indication-deep-dive, compare mode, temporal tracking. Premium users pay for full API surface.

The Outsider

Premium feature doesn't check subscription upfront — 30+ seconds wasted. File naming convention undocumented — silent missing sections. 500-char truncation serves developer convenience, not user workflow. Works mechanically but not experientially.

The Executor

3 P0 fixes (~2 hours): (1) Parameterize truncation (3-line change), (2) Document section ID extraction, (3) Subscription check at Step 1. P1: fix load_json, manifest, cross-skill hints. P2: CSV, multimedia, compare mode.

Peer Review Highlights

Key insights:

  • User task definition missing — triage vs analysis determines whether 500 chars is fine or broken
  • Cascade failure if Step 2 fails silently
  • No testability without live premium account
  • Output destination unspecified (terminal vs file vs LLM pipeline)
  • No observability — 30-second silent process looks like a hang
