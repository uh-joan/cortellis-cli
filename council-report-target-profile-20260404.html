LLM Council Verdict

Target Profile Skill Production Readiness — April 4, 2026

Question

Is the cli_anything/cortellis/skills/target-profile/ skill complete enough for production use? Are there meaningful gaps?

Council Verdict Not Production-Ready

The 7-dimension architecture is correct and the resolution strategy is reasonable. But the skill systematically misrepresents its own completeness. SKILL.md says "ALWAYS list ALL items — no truncation" while the code silently truncates every dimension. The drug pipeline shows 50 rows under a header that says "847 total." A scientist cannot trust this output without already knowing its limitations — which inverts the value proposition.

Do not ship in production, including internally without explicit caveats.

Internal Contradiction (flagged by Outsider)

SKILL.md documentation says "ALWAYS list ALL items in tables (no truncation)". The code does: disease_data[:25], gene_assocs[:15], meaningful[:20], pharm_list[:20], and drug pipeline --hits 50. The report header shows the real @totalResults while the table only shows the truncated subset — making the gap visible to careful readers but unexplained.

Advisor Alignment

Unanimous Agreement

  • 50-hit pipeline cap is the most damaging defect
  • Silent truncation across ALL dimensions (25/15/20/20)
  • Cross-skill dependency is undocumented coupling
  • No coverage/confidence signals in output
  • Sequential API calls waste 60-70% of wall time

Key Disagreements

  • Executor: resolution is "solid" / Outsider: hard stop, no fuzzy fallback
  • Executor: disease briefings fail "gracefully" / Others: silent data hole
  • Expansionist: add /target-compare / Others: fix single target first

Gaps Identified (by severity)

  • Drug pipeline capped at 50, no pagination. EGFR has 300+ programs. Header shows totalResults (847) but table has 50 rows. This is a lie of omission — the number signals completeness, the data doesn't deliver.
  • SKILL.md contradicts code behavior. Documentation promises "no truncation" while code hard-caps every dimension. Users have been told a false contract.
  • No "Showing X of Y" on any truncated section. A researcher sees 20 protein interactions with no idea if that's 20 of 20 or 20 of 300.
  • Resolution failure propagates silently. Wrong target ID at Step 1 produces plausible-looking but incorrect data across all 7 dimensions. No validation layer between resolution and retrieval.
  • Cross-skill dependency via relative path. If landscape skill is absent, action resolution returns empty string and drug pipeline (Step 6) is silently skipped.
  • Protein interaction filter is undisclosed. Names >60 chars filtered silently. Reported count may not match table row count.
  • No output provenance. No run timestamp, no API endpoint attribution per dimension. Two runs produce different results with no record of when data was fetched.
  • Disease briefings fail silently. 400 errors cause the section to be omitted with no indication to the user.
  • Sequential API calls. 7-8 independent calls run sequentially. Parallelism would cut wall time 60-70%.

The One Thing to Do First

Fix drug pipeline pagination (paginate until exhausted or hit a reasonable cap) and add "Showing X of Y" to every truncated list — in the same commit, add a run timestamp to the output header. This closes the header/table contradiction, makes truncation visible across all dimensions, and begins the provenance record.

Advisor Responses

Contrarian "Demo-quality that looks production-ready but hides incompleteness"

Same fatal flaw as landscape, worse here. EGFR has 300+ drugs, skill shows 50 (15% coverage). Cross-skill dependency is production anti-pattern — no contract, no versioning, no fallback. 7-8 sequential calls unusable for batch. Disease briefings "optional, may fail" not acceptable. Demo-quality that produces reports which look production-ready but hide their own incompleteness.

First Principles "Production readiness requires outputs trustworthy by default, not by user vigilance"

Wrong question. Real question: can a scientist trust outputs without knowing limitations? No. Silent truncation is epistemological failure — 50 for EGFR (200+) gives data that looks complete. No pagination is unvalidated assumption. Cross-skill dependency is undocumented coupling. Sequential calls are structurally slow for no reason. Fix the epistemic layer first.

Expansionist "Adjacent opportunity: cross-target comparison (/target-compare)"

50-hit cap most damaging. Display limits are hard truncations with no signal. Natural extension: /target-compare for druggability triage. Resolution confidence should surface in header. Disease briefings 400s need monitoring. Fix pagination and coverage signals — same landscape lesson.

Note: Council consensus is to fix single-target integrity before expanding to comparison.

Peer Review Highlights

Unanimous strongest: The Outsider — grounded in code evidence, not principles. Cited specific slice notation and the SKILL.md contradiction.

Unanimous weakest: The Executor — called resolution "solid" and disease briefings "graceful" without evidence. "Ship internal" is not a meaningful risk boundary.

All advisors missed:

  • Output provenance — no timestamps, no run metadata. Two runs produce different results with no record of when data was fetched. Traceability failure for scientific use.
  • Error propagation across dimensions — wrong target ID at resolution produces plausible-looking but incorrect data in all 7 dimensions. No validation layer. Worse than a hard stop.
  • Data freshness — no indication of when underlying Cortellis data was last updated. Stale pipeline counts are patient-safety-adjacent errors.
