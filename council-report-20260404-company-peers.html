2026-04-04 · Company Peers Skill Assessment · 5 advisors, 5 peer reviews

Is the /company-peers skill in the Cortellis CLI complete enough for production use? What gaps exist?

The Recommendation: Do Not Ship

The skill has a confirmed broken dependency (always-empty peer table), silent failure modes, and a scope limitation that would mislead strategic decision-making. Shipping it would erode trust faster than not shipping it.

The One Thing to Do First

Fix the schema contract between find_peers.py and the report generator.

find_peers.py writes peers.json (flat array with name, drug_count, overlap). The report reads peers_ind*.json expecting Cortellis company search response schema (@id, @name, Country). Incompatible filename AND schema — peer table is always empty.

Rewrite the report to read peers.json directly. One-file change. Until this works, nothing else matters.

Fix Roadmap

Phase 1 — Make It Work

  • Fix filename/schema mismatch between find_peers.py and report generator
  • Replace bare except Exception: return [] with typed exceptions + stderr
  • Add credential pre-flight check
  • Document Step 4 dependency (Step 5 needs peer IDs from Step 4)

Phase 2 — Make It Credible

  • Expand peer search beyond Launched to include Phase 2/3 pipeline companies
  • Document selection methodology in report output
  • Surface data timestamp for freshness
  • Add cross-skill navigation hints (/head-to-head, /pipeline)

Blind Spots the Council Caught

No Schema Contract

No shared contract between find_peers.py and the report generator. Any change to either silently breaks the pipeline. No test fixtures means regressions go undetected.

Methodology Opaque to Users

"Top 10 peers" with no explanation of selection criteria is uninterpretable. Users cannot evaluate whether the peer set is appropriate for their question.

No Observability

No logging, no run IDs, no intermediate artifact validation. When the report is wrong, there is no audit trail for reproducibility.

Data Freshness Unaddressed

Cortellis data has approval lag. Launched-only filter on stale data compounds the bias.

Advisor Responses

The Contrarian

Three blockers: (1) bare except Exception: return [] silently swallows all errors. (2) Report reads peers_ind*.json but SKILL.md creates peers.json — broken dependency. (3) Launched-only peer search excludes Phase 3 competitors — strategically misleading.

Also: no manifest, no credential check, no cross-skill hints. Step 4 "optional" with no criteria creates inconsistent outputs.

The First Principles Thinker

"What makes someone a real competitor?" Current answer: "who else has launched drugs in same indications" — a proxy, not the thing.

Five gaps: Launched-only bias, indication matching too narrow (vs mechanism overlap), metrics show size not positioning, no stage-weighting, company not positioned within its own peer set.

The Expansionist

Missing: forward-looking signals (trial momentum), technology platform intelligence, network effects (partnerships), deal terms (deals-intelligence), emerging peers (Phase 2 companies = future competitors + acquisition targets).

The Outsider

Only approved drugs checked — most dangerous competitors not yet approved. "Top 10" arbitrary. Success rate undefined. No financial dimension. Head-to-head depends on this tool's selection but criteria too narrow.

The Executor

BUG 1: find_peers.py writes flat JSON to peers.json but report reads peers_ind*.json with company search schema — incompatible. Peer table always empty.

BUG 2: Schema mismatch — find_peers returns (name, drug_count, overlap) but report expects (@id, @name, Country, Drugs.@activeDevelopment).

Step 4 "optional" but Step 5 needs peer IDs from it. Monday fix: rewrite report to read peers.json directly.

Peer Review Highlights
