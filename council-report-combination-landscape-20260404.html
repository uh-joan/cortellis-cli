LLM Council Verdict

Combination Landscape Skill Production Readiness — April 4, 2026

Question

Is the cli_anything/cortellis/skills/combination-landscape/ skill complete enough for production use? Are there meaningful gaps?

Council Verdict Not Production-Ready

The skill is a working prototype with correct scaffolding but incorrect core logic. The infrastructure (report generator, table formatting, CLI wiring) is sound. The data retrieval strategy is not.

Searching for "+" in drug names catches branded co-formulations but misses the investigational combinations that matter most in oncology (FOLFOX, pembrolizumab+olaparib, nivolumab+ipilimumab). This is not a truncation issue — it is the wrong query for the right problem.

Do not ship to oncology/immunology users in this state.

Key Question (raised by peer review)

Does the Cortellis API have a dedicated combination therapy endpoint or structured combination flag? If yes, rewrite search logic around it. If no, the precision ceiling is inherent and must be documented explicitly. Every other fix is downstream of this answer.

Advisor Alignment

Unanimous Agreement

  • No pagination (50-hit cap on both searches)
  • No "Showing X of Y" or coverage signals
  • Silent truncation at 30 drugs / 20 trials
  • Cross-dependency on landscape not inlined

Key Disagreement

  • Contrarian/FP: search logic is broken by design
  • Executor: 20 lines of Python fixes it
  • Verdict: Executor underestimates — fixing pagination on a broken query is waste

Gaps Identified (by severity)

  • Search logic misses most real combinations. "+" in drug name catches branded co-formulations only. FOLFOX has no "+". Investigational combos (pembrolizumab+olaparib) are not branded. The query returns a biased sample of the wrong population.
  • "combination" trial keyword is noise. Full-text match returns trials mentioning "combination" anywhere. Misses trials where combination is implicit in design. No structured co-enrollment filter.
  • No pagination on either search. --hits 50 cap. Oncology indications have hundreds of combination programs.
  • Silent truncation at 30/20 display limits. No "Showing X of Y". User sees clean table with no indication of what's missing.
  • Components column missing. SKILL.md specifies a Components column. The report generator never extracts or renders it. The most diagnostic field for combinations is absent.
  • ActionsPrimary dict rendering bug. Can render as {'$': 'inhibitor'} in table cells instead of the actual mechanism name.
  • Cross-dependency not inlined. Depends on landscape's resolve_indication.py. Same fragility fixed in target-profile but not here.
  • Fallback logic is prose, not code. SKILL.md says "if 0 results, try Co-formulation" but no recipe enforces this programmatically.
  • No coverage footer. @totalResults is read but never surfaced to user.

The One Thing to Do First

Determine whether the Cortellis API exposes a combination therapy endpoint or a structured combination flag on drug records. If yes, rewrite the search logic around that endpoint. If no, document the precision ceiling explicitly in the skill output. Every other fix (pagination, coverage signals, Components column) is downstream of this answer.

Advisor Responses

Contrarian "Confidently wrong outputs with no signal that anything is missing"

Primary detection method broken by design — "+" in drug names catches branded co-formulations but misses investigational combos. "combination" as trial keyword is noise. Cross-dependency fragile. No pagination. Will produce confidently wrong outputs.

First Principles "Search logic is wrong by design — don't ship to oncology users" — Voted strongest by 2 of 3 reviewers

"+" catches only co-formulated branded products, not actual regimens (FOLFOX, carboplatin+pembrolizumab). Components column missing despite SKILL.md spec. Trials keyword not structured. Most thorough technical critique with spec violation evidence.

Expansionist "Highest-signal query type — two hours work for disproportionate value"

Three deficiencies but combination therapy is highest-signal query in Cortellis. Fix is same playbook as landscape. Phase 2→3 transitions are densest here and BD deal flow concentrates around combos.

Outsider "Same bug class across codebase — treat as systematic, not per-skill"

Silent truncation critical. Search logic fragile. Dependency undocumented risk. Calls out the pattern: this codebase has a habit of silent truncation that should be treated as a bug class, not individual per-skill issues.

Executor "Three concrete blockers — 20 lines of Python"

No X of Y, no footer, no pagination. Found secondary dict rendering bug. But peer reviews challenged the "20 lines" estimate — fixing pagination on a broken query doesn't fix the query.

Peer Review Highlights

Strongest: First Principles (2 votes) — root cause analysis + spec violation evidence.

All advisors missed:

  • API capability question — does Cortellis even have a dedicated combination endpoint? If not, the skill is reverse-engineering structured data from unstructured text and precision will always be low.
  • No empirical testing — all critiques are code-reading. A 10-minute smoke test would confirm or refute whether "+" detection actually misses combos in practice.
  • Cross-dependency failure mode — nobody examined whether it's a subprocess, import, or config coupling. The blast radius of failure is unknown.
