LLM Council Verdict

Drug Comparison Skill Production Readiness — April 4, 2026

Question

Is the cli_anything/cortellis/skills/drug-comparison/ skill complete enough for production use?

Council Verdict Not Production-Ready

The architecture is sound and the comparison logic is well-built. But a SWOT data collapse bug (only first entry rendered per category) combined with silent truncation across trials, deals, and CSV export means the skill can produce confidently incomplete comparisons with no visible signal.

Fix surface is small (~30 min). The skill is close.

Bug Found (Outsider, verified in code)

SWOT section uses items[0] — only the first strength/weakness/opportunity/threat is ever rendered per category. Multiple entries collapse to one with no warning. This is a data loss bug, not a truncation design choice.

Advisor Alignment

Unanimous Agreement

  • Architecture is sound, dict handling excellent
  • Trials/deals need "Showing X of Y"
  • No timestamp = hard gap for deal-making
  • SWOT truncation is a real problem

Key Disagreement

  • Executor: SWOT cap is "intentional for readability"
  • Outsider: it's items[0] collapse — a bug
  • Expansionist: timestamp is "non-issue"
  • FP/Contrarian: timestamp is hard gap for deals

Gaps Identified

  • SWOT items[0] collapse. Only first entry per SWOT category rendered. Multiple strengths/weaknesses lost silently. Data loss bug.
  • Trials: no "Showing X of Y". Total shown in header but [:10] cap silent. Trial volume IS a competitive signal (847 vs 12).
  • Deals: same pattern. Total shown but display silently capped at 10.
  • No timestamp. Hard gap for deal-making contexts where data currency matters.
  • CSV export: limit=5 silent. Export is the artifact most likely to leave the tool and land in a deck.
  • No coverage footer. No summary of what data was available vs missing per drug.
  • Resolver SPOF. Wrong drug ID = confidently wrong comparison across all 8 dimensions. Surface resolved name+ID per column.

The One Thing to Do First

Fix the items[0] SWOT collapse to iterate over all entries, and add resolver confirmation headers (resolved drug name + ID per column). These address the only failure modes where the skill can be confidently wrong without any visible signal.

Contrarian "Silent truncation is systematic failure at the skill's one job"

4 defects: SWOT 150-char truncation misleads, CSV limit=5 broken for bulk use, trials/deals lack "Showing X of Y" (trial volume IS a signal), no timestamp. Fix those four, then production-ready.

First Principles "Comparison has different requirements — 'complete data per drug?' not 'find all drugs?'"

Timestamp is hard gap. "Showing 10 of 47" needed. Missing data vs no data indistinguishable. Resolver is SPOF. Architecture sound, gap is transparency.

Expansionist "SWOT truncation kills signal — raise to 400+ or bulleted sub-section"

3 gaps: SWOT truncation, indication overlap hardcoded for 2 drugs, no Verdict synthesis. Coverage footer "non-issue." Architecture is right.

Outsider "SWOT shows only items[0] — multiple entries collapse to one" — Voted strongest

Verified in code: trials [:10] no warning, deals [:10] no warning, SWOT items[0] collapse, At-a-Glance [:5] no note, CSV limit=5 silent. The only advisor who read the actual code.

Executor "6 fixes, ~30 min — architecture solid"

SWOT truncation "intentional" (peer review disagrees). Per-drug "Showing X of Y". Dynamic N-drug footer. Gaps are cosmetic. Note: peer review challenged the "intentional" claim.

Peer Review Highlights

Strongest: Outsider — only one that verified against code.

All missed: Resolver is the real SPOF. Wrong drug ID = confidently wrong comparison across all dimensions with no signal. Silent wrong data is categorically worse than visible truncation.
