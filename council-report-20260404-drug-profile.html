LLM Council Report

"Is the /drug-profile skill complete enough for production use, or does it have meaningful gaps?"
5 advisors • 3 peer reviews • 1 chairman synthesis • April 4, 2026
NOT PRODUCTION-READY

The skill produces reports that look authoritative but silently omit data, truncate against their own stated rules, ignore fetched competitor pipelines, discard drug design enrichment, and misrepresent regulatory documents as approval history.

The bones are solid (resolver, SWOT delegation, 10-endpoint scope). The problems are all in the last mile — how collected data becomes the output report. Fixable without rearchitecting.

Where the Council Agrees

  • Truncation contradicts the skill's own rules. SKILL.md says "ALWAYS list ALL items. No truncation." Code hardcodes [:10], [:15], [:5] limits. Headers show totals like "Deals (47 total)" while tables show 10. Four of five advisors flagged this independently.
  • The competitive landscape is blind to the pipeline. Only launched drugs shown. Phase 3 competitor data is fetched (competitors_p3.json) but never rendered. For early-stage drugs: zero competitive context.
  • Drug Design enrichment is dead code. Step 10 runs, writes files, report generator never reads them. The Executor caught this; the Expansionist praised it as "brilliant" without verifying.

Where the Council Clashes

  • Is this one skill or three? First Principles says split by use case (deal eval, positioning, investment). Executor/Contrarian say fix the shared foundation first. Pragmatic answer: fix the common problems before considering a split.
  • Are limits bugs or features? Expansionist frames caps as future features. Everyone else: when your docs promise no truncation and your code truncates, that's a bug. Peer reviews unanimously sided against the Expansionist.

Blind Spots the Council Caught

  • Silent partial failure is the worst outcome. All three peer reviews converged: load_json uses bare except: that returns None. When API calls fail, user receives a report that looks complete but is missing sections. No error logging, no incompleteness signal. More dangerous than a hard failure.
  • The regulatory query is semantically wrong. The Contrarian caught that regulations search returns documents/news, not drug approval status. Section heading implies approval landscape; data is press clippings.

The Recommendation

Do not ship this for production decision-making. A confident-looking but incomplete report is actively harmful for deal evaluation and investment decisions. The problems are all in the last mile — fixable without rearchitecting.

The One Thing to Do First

Replace the bare except: in load_json with proper error logging and add a "Data Completeness" header to the report output that lists which API calls succeeded and which failed. A partial report labeled as partial is safe. A partial report labeled as complete is a liability.

Advisor Responses

Contrarian "Built for a demo, not production"

1. Truncation lies. Output rule says "ALWAYS list ALL items. No truncation." Code hardcodes [:10] everywhere. For semaglutide with dozens of trials, a user gets 10 rows and no indication they're missing the rest.

2. Regulatory section is mislabeled. Queries regulations search (documents/news endpoint), not drug approval status. Someone reads "3 regulatory documents" and thinks they understand the approval landscape. They won't.

3. Financial section truncates to 500 chars of prose. Actual revenue figures are in structured fields this skill never touches. Rules say "give exact numbers" — delivers a truncated paragraph.

Resolver is solid. SWOT delegation is reasonable. Report generator was built for a demo.

First Principles "Three skills wearing one coat"

Drug profiles serve three use cases requiring different data: deal evaluation (IP cliffs, peak sales threats), competitive positioning (Phase 1/2 pipeline), investment decisions (regulatory risk, trial patterns).

The current implementation fails at all three: financials truncated at 500 chars (fatal for deals), competitors limited to launched (fatal for positioning), history drops approvals/trials (fatal for investment).

Fix the use-case split first. Then gaps become obvious and bounded.

Expansionist "Under-exploiting its own infrastructure"

Every limit is a feature request. Deals at 10 should be 50+ with deal-type filtering. Drug Design enrichment is accidentally brilliant — expand to all drugs. The 21 adjacent skills should auto-cross-link into a living intelligence graph.

Production-ready for basic use. Dramatically under-exploiting for power users.

Outsider "These cause wrong calls"

1. Silent truncation. Headers show totals but tables show 10. SKILL.md contradicts the code.

2. Competitive landscape too narrow. Only launched drugs. Phase 3 data fetched but never rendered. Drug looks like a leader while facing five hidden challengers.

3. Timeline nearly empty. Only phase transitions. A 15-year drug shows 4 lines.

Executor "Monday morning fix order provided"

5 blockers: (1) bare except: silences errors, (2) timeline drops approvals/safety events, (3) 500-char financial truncation, (4) Drug Design files written but never read, (5) launched-only competitors.

Fix order: error logging → timeline expansion → Drug Design wiring → Phase 3 competitors.

Peer Review Highlights

Strongest advisor: Executor (2 votes) and Contrarian (1 vote). Executor uniquely caught Drug Design dead code and provided prioritized fix order.

Weakest advisor: Expansionist (unanimous). Treated production bugs as feature requests. Praised Drug Design without verifying it's consumed.

All missed: API failure handling. What happens when 3 of 10 calls fail? Silent partial reports are worse than hard failures. Critical for downstream automation.
